# S3 Server Autoloop

Codex-driven autonomous loop for completing the `devcloud` S3-compatible object storage server.

## Start

```bash
bash scripts/s3-autoloop/run-loop.sh
```

The default verification gate is `foundation`, so the runner can start from the current Mail-only repository and advance one implementation phase at a time. A `DONE` footer from Codex is accepted only after the full S3 gate passes.

## Useful Options

```bash
MAX_ITERATIONS=20 bash scripts/s3-autoloop/run-loop.sh
AUTOCOMMIT=true bash scripts/s3-autoloop/run-loop.sh
CODEX_ARGS="--full-auto" bash scripts/s3-autoloop/run-loop.sh
VERIFY_STAGE=foundation bash scripts/s3-autoloop/run-loop.sh
DONE_VERIFY_STAGE=full bash scripts/s3-autoloop/run-loop.sh
VERIFY_STAGE=full bash scripts/s3-autoloop/verify.sh
```

## Verification Stages

- `foundation`: Go tests, CLI help, daemon help, and `cmd/devcloud` build.
- `config`: validates S3 config shape, workspace initialization, and storage directories.
- `s3-core`: starts `devcloud up` and checks S3 endpoint, bucket CRUD, object CRUD, metadata, range reads, and dashboard availability.
- `s3-multipart`: checks multipart upload create, upload part, list parts, complete, abort, and list uploads.
- `dashboard-static`: checks S3 dashboard/API visibility without requiring a React runtime.
- `hardening` / `full`: final acceptance gate for the S3 MVP.

## Files

- `goal.md`: measurable implementation contract.
- `bootstrap.sh`: initializes resumable state and validates shell syntax.
- `run-loop.sh`: Codex runner with bounded iterations, lock, status parsing, verification gates, and checkpoint writes.
- `verify.sh`: acceptance verification gate.
- `recover.sh`: rebuilds `state.env` from `progress.md` and clears stale lock/circuit state.
- `progress.md`: append-only loop timeline generated by the runner.
- `state.env`: resumable execution state generated by the runner.

The runner expects Codex CLI to be available as `codex`.
